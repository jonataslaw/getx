# Get کے بارے میں

- GetX Flutter کے لیے ایک ہلکا اور طاقتور حل ہے۔ یہ ہائی پرفارمنس اسٹیٹ مینجمنٹ، اسمارٹ ڈیپنڈنسی انجیکشن، اور روٹ مینجمنٹ کو تیزی اور آسانی سے یکجا کرتا ہے۔

- GetX کے تین بنیادی اصول ہیں: **پروڈکٹیویٹی، پرفارمنس اور آرگنائزیشن**۔

  - **پرفارمنس:** GetX پرفارمنس اور کم سے کم وسائل کے استعمال پر مرکوز ہے۔ یہ Streams یا ChangeNotifier استعمال نہیں کرتا۔

  - **پروڈکٹیویٹی:** GetX آسان اور خوشگوار syntax استعمال کرتا ہے۔ جو بھی آپ کرنا چاہتے ہیں، GetX کے ساتھ ہمیشہ آسان طریقہ موجود ہے۔ یہ آپ کے development کے گھنٹے بچائے گا اور آپ کی ایپلیکیشن کی زیادہ سے زیادہ پرفارمنس دے گا۔

    عام طور پر، ڈویلپر کو کنٹرولرز کو میموری سے ہٹانے کی فکر ہوتی ہے۔ GetX کے ساتھ یہ ضروری نہیں کیونکہ resources خود بخود میموری سے ہٹا دیے جاتے ہیں جب وہ استعمال میں نہ ہوں۔ اگر آپ انہیں میموری میں رکھنا چاہتے ہیں تو "permanent: true" declare کریں۔ اس طرح آپ کا وقت بھی بچے گا اور غیر ضروری dependencies میموری میں نہیں رہیں گی۔ Dependency loading بھی lazy by default ہے۔

  - **آرگنائزیشن:** GetX آپ کو View، presentation logic، business logic، dependency injection اور navigation کو مکمل طور پر الگ کرنے کی اجازت دیتا ہے۔ آپ کو routes کے درمیان navigate کرنے کے لیے context کی ضرورت نہیں، اس لیے آپ widget tree پر منحصر نہیں ہیں۔ آپ کو inheritedWidget کے ذریعے اپنے controllers/blocs تک رسائی کے لیے context کی ضرورت نہیں، اس لیے آپ کی presentation logic اور business logic visualization layer سے مکمل طور پر الگ ہو جاتی ہے۔ آپ کو اپنے Controllers/Models/Blocs کو widget tree میں inject کرنے کی ضرورت نہیں۔ اس کے لیے GetX اپنا dependency injection feature استعمال کرتا ہے۔

    GetX کے ساتھ آپ کو اپنی ایپلیکیشن کی ہر feature کہاں ملے گی، معلوم ہوتا ہے، اور کوڈ صاف رہتا ہے۔ اس سے maintenance آسان ہو جاتی ہے اور modules کا اشتراک بھی ممکن ہے۔ BLoC نے Flutter میں کوڈ organize کرنے کی بنیاد رکھی، اس نے business logic کو visualization سے الگ کیا۔ GetX اس کی قدرتی ترقی ہے، جو نہ صرف business logic بلکہ presentation logic کو بھی الگ کرتا ہے۔ Dependency injection اور routes بھی الگ ہیں، اور data layer بھی الگ ہے۔ سب کچھ آسان طریقے سے۔
    GetX Flutter SDK کے ساتھ ہائی پرفارمنس ایپلیکیشنز بنانے کا آسان، عملی اور scalable طریقہ ہے۔ اس کے ارد گرد بڑا ecosystem ہے، یہ beginners کے لیے آسان اور experts کے لیے درست ہے۔ یہ محفوظ، مستحکم، up-to-date ہے اور بہت سی built-in APIs فراہم کرتا ہے جو default Flutter SDK میں نہیں ہیں۔

- GetX bloated نہیں ہے۔ اس میں بہت سی features ہیں لیکن ہر feature الگ container میں ہے اور صرف استعمال پر شروع ہوتی ہے۔ اگر آپ صرف State Management استعمال کرتے ہیں تو صرف وہی compile ہوگی۔

- GetX کا بڑا ecosystem، community اور contributors ہیں اور یہ Flutter کے ساتھ maintained رہے گا۔ GetX ایک ہی کوڈ کے ساتھ Android، iOS، Web، Mac، Linux، Windows اور server پر چل سکتا ہے۔
  **آپ اپنا frontend کا کوڈ backend پر بھی [Get Server](https://github.com/jonataslaw/get_server) کے ساتھ reuse کر سکتے ہیں۔**

**اس کے علاوہ، پورا development process server اور frontend دونوں پر [Get CLI](https://github.com/jonataslaw/get_cli) کے ساتھ automate کیا جا سکتا ہے۔**

**مزید productivity کے لیے ہمارے پاس
[VSCode extension](https://marketplace.visualstudio.com/items?itemName=get-snippets.get-snippets) اور [Android Studio/Intellij extension](https://plugins.jetbrains.com/plugin/14975-getx-snippets) بھی ہیں۔**

# GetX کے ساتھ کاؤنٹر ایپ

Flutter میں default "counter" project 100 سے زیادہ لائنز پر مشتمل ہے۔ Get کی طاقت دکھانے کے لیے، یہاں صرف 26 لائنز میں ایک "counter" ایپ بنائیں گے جو state کو ہر کلک پر بدلتی ہے، صفحات کے درمیان state share کرتی ہے، اور business logic کو view سے الگ رکھتی ہے۔

- مرحلہ 1:
  اپنے MaterialApp سے پہلے "Get" شامل کریں، اسے GetMaterialApp میں بدل دیں:

```dart
void main() => runApp(GetMaterialApp(home: Home()));
```

- نوٹ: یہ Flutter کے MaterialApp کو modify نہیں کرتا، GetMaterialApp صرف ایک pre-configured Widget ہے۔ اگر آپ صرف state management یا dependency management استعمال کر رہے ہیں تو GetMaterialApp ضروری نہیں۔ یہ routes، snackbars، internationalization، bottomSheets، dialogs وغیرہ کے لیے ضروری ہے۔
- نوٹ²: یہ step صرف route management (`Get.to()`, `Get.back()` وغیرہ) کے لیے ضروری ہے۔

- مرحلہ 2:
  اپنی business logic class بنائیں اور تمام variables، methods اور controllers اس میں رکھیں۔ کسی بھی variable کو ".obs" لگا کر observable بنا سکتے ہیں۔

```dart
class Controller extends GetxController{
  var count = 0.obs;
  increment() => count++;
}
```

- مرحلہ 3:
  اپنی View بنائیں، StatelessWidget استعمال کریں اور RAM بچائیں۔ Get کے ساتھ StatefulWidget کی ضرورت کم ہو جاتی ہے۔

```dart
class Home extends StatelessWidget {

  @override
  Widget build(context) {

    // اپنی class کو Get.put() سے instantiate کریں تاکہ یہ تمام child routes کے لیے available ہو جائے۔
    final Controller c = Get.put(Controller());

    return Scaffold(
      // Obx(()=> استعمال کریں تاکہ Text() ہر بار count بدلنے پر update ہو جائے۔
      appBar: AppBar(title: Obx(() => Text("Clicks: ${c.count}"))),

      // 8 لائنز Navigator.push کی جگہ صرف Get.to() استعمال کریں۔ context کی ضرورت نہیں۔
      body: Center(child: ElevatedButton(
              child: Text("Go to Other"), onPressed: () => Get.to(Other()))),
      floatingActionButton:
          FloatingActionButton(child: Icon(Icons.add), onPressed: c.increment));
  }
}

class Other extends StatelessWidget {
  // Get سے وہ Controller تلاش کریں جو کسی اور صفحہ پر استعمال ہو رہا ہے۔
  final Controller c = Get.find();

  @override
  Widget build(context){
     // updated count variable تک رسائی
     return Scaffold(body: Center(child: Text("${c.count}")));
  }
}
```

یہ ایک سادہ project ہے لیکن یہ Get کی طاقت کو واضح کرتا ہے۔ جیسے جیسے آپ کا project بڑھے گا، یہ فرق اور نمایاں ہو جائے گا۔

Get ٹیمز کے ساتھ کام کرنے کے لیے ڈیزائن کیا گیا ہے، لیکن یہ انفرادی ڈویلپر کے لیے بھی آسان ہے۔

اپنی deadlines بہتر بنائیں، سب کچھ وقت پر deliver کریں بغیر پرفارمنس کھوئے۔ اگر آپ کو یہ سب پسند آیا تو Get آپ کے لیے ہے!

# تین ستون

## اسٹیٹ مینجمنٹ

Get کے پاس دو مختلف state managers ہیں: simple state manager (GetBuilder) اور reactive state manager (GetX/Obx)

### ری ایکٹو اسٹیٹ منیجر

ری ایکٹو programming کو اکثر مشکل سمجھا جاتا ہے۔ GetX اسے بہت آسان بنا دیتا ہے:

- آپ کو StreamControllers بنانے کی ضرورت نہیں۔
- ہر variable کے لیے StreamBuilder بنانے کی ضرورت نہیں۔
- ہر state کے لیے الگ class بنانے کی ضرورت نہیں۔
- initial value کے لیے get بنانے کی ضرورت نہیں۔
- code generators کی ضرورت نہیں۔

Get کے ساتھ reactive programming اتنی ہی آسان ہے جتنا setState استعمال کرنا۔

فرض کریں آپ کے پاس ایک name variable ہے اور آپ چاہتے ہیں کہ جب بھی یہ بدلے، تمام widgets خود بخود update ہو جائیں۔

یہ ہے آپ کا count variable:

```dart
var name = 'Jonatas Borges';
```

اسے observable بنانے کے لیے صرف ".obs" لگا دیں:

```dart
var name = 'Jonatas Borges'.obs;
```

اور UI میں، جب آپ یہ value دکھانا چاہیں اور ہر بار value بدلنے پر screen update ہو، تو یوں کریں:

```dart
Obx(() => Text("${controller.name}"));
```

بس اتنا ہی۔ یہ _اتنا_ آسان ہے۔

### اسٹیٹ مینجمنٹ کی مزید تفصیل

**اسٹیٹ مینجمنٹ کی مزید وضاحت [یہاں](./documentation/en_US/state_management.md) دیکھیں۔ وہاں مزید مثالیں اور simple state manager اور reactive state manager میں فرق بھی ملے گا۔**

آپ کو GetX کی طاقت کا اچھا اندازہ ہو جائے گا۔

## روٹ مینجمنٹ

اگر آپ routes/snackbars/dialogs/bottomsheets بغیر context کے استعمال کرنا چاہتے ہیں تو GetX آپ کے لیے بہترین ہے:

"Get" کو اپنے MaterialApp سے پہلے شامل کریں، اسے GetMaterialApp میں بدل دیں:

```dart
GetMaterialApp( // پہلے: MaterialApp(
  home: MyHome(),
)
```

نئی screen پر جائیں:

```dart
Get.to(NextScreen());
```

نام کے ساتھ نئی screen پر جائیں۔ named routes کی مزید تفصیل [یہاں](./documentation/en_US/route_management.md#navigation-with-named-routes)

```dart
Get.toNamed('/details');
```

snackbars، dialogs، bottomsheets یا کوئی بھی چیز جو Navigator.pop(context) سے بند ہوتی ہے، اسے بند کریں:

```dart
Get.back();
```

اگلی screen پر جائیں اور پچھلی screen پر واپس جانے کا option نہ ہو (مثلاً SplashScreens، login screens وغیرہ میں):

```dart
Get.off(NextScreen());
```

اگلی screen پر جائیں اور تمام پچھلے routes cancel کر دیں (shopping carts، polls، tests وغیرہ میں مفید):

```dart
Get.offAll(NextScreen());
```

دیکھا آپ کو کسی بھی چیز کے لیے context استعمال نہیں کرنا پڑا؟ یہ Get route management کا بڑا فائدہ ہے۔ اس سے آپ یہ سب methods اپنے controller class میں بھی استعمال کر سکتے ہیں۔

### روٹ مینجمنٹ کی مزید تفصیل

**Get named routes کے ساتھ کام کرتا ہے اور routes پر lower-level control بھی دیتا ہے! مکمل documentation [یہاں](./documentation/en_US/route_management.md)**

## ڈیپنڈنسی مینجمنٹ

Get کے پاس ایک سادہ اور طاقتور dependency manager ہے جو آپ کو صرف 1 لائن میں اپنی class (جیسے Bloc یا Controller) حاصل کرنے دیتا ہے، نہ Provider context، نہ inheritedWidget:

```dart
Controller controller = Get.put(Controller()); // بجائے Controller controller = Controller(); 
```

- نوٹ: اگر آپ Get کا State Manager استعمال کر رہے ہیں تو bindings API پر زیادہ توجہ دیں، یہ آپ کی view کو controller سے جوڑنا آسان بنا دے گا۔

اپنی class کو اس class میں instantiate کرنے کے بجائے جس میں آپ استعمال کر رہے ہیں، آپ اسے Get instance میں instantiate کرتے ہیں، جو اسے پوری App میں available کر دیتا ہے۔ اس طرح آپ اپنے controller (یا Bloc class) کو عام طریقے سے استعمال کر سکتے ہیں۔

**ٹپ:** Get dependency management دوسرے حصوں سے الگ ہے، اس لیے اگر آپ کی app پہلے ہی کوئی state manager استعمال کر رہی ہے تو بھی آپ یہ dependency injection بغیر کسی مسئلے کے استعمال کر سکتے ہیں۔

```dart
controller.fetchApi();
```
